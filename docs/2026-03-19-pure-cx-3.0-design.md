# 纯 CX 3.0 设计稿

日期: 2026-03-19

状态: Draft

作者: chengxuan

## 1. 背景

当前 `cx-workflow` 已经形成两套同时存在的使用形态:

- 插件仓库中的通用 `cx` 内核
- 项目中围绕 `.claude/cx` 演化出来的运行时结构与使用习惯

与此同时，历史上的 `tc` 命令层提供了不少高频、轻量、顺手的体验，但这些体验并没有系统性回流到 `cx` 插件层，导致出现以下问题:

- 插件文档与项目真实运行状态存在版本分叉，典型表现是 `references/config-schema.json` 仍以 `2.0` 叙事为主，而项目运行配置已经朝 `3.0` 演进
- `cx` 与 `tc` 在命令职责和使用心智上存在重复
- 项目级 `.claude/cx` 已经成为事实上的运行时真相，但插件文档对这一点定义不够明确
- 轻量体验缺少正式承接路径，容易继续依赖本地习惯而不是插件协议

本设计稿的目标，是将未来工作流正式收敛为一套纯 `cx` 体系。

## 2. 目标

- 停止发展 `tc` 命令面，未来只保留 `cx-*` 命令
- 以项目级 `.claude/cx` 作为唯一运行时真相
- 将 `tc` 中高频、轻量、顺手的能力并入现有 `cx` 命令内部的自动路由
- 将插件仓库重新定义为“能力内核”，而不是项目状态存储层
- 将 `cx` 的公开配置、状态协议、命令语义、hook 行为统一提升到 `3.0` 版本叙事
- 保持默认自动化、低打断、中文友好

## 3. 非目标

- 不做 `tc` 到 `cx` 的兼容层保留
- 不在本次设计中引入新的平行命令集
- 不把 GitHub Issue 体系作为运行时主控真相
- 不把项目运行状态重新挪回插件用户目录
- 不追求“所有路径都必须最重最完整”的流程化设计

## 4. 总体原则

未来纯 `cx 3.0` 遵循以下原则:

- 插件层提供能力，项目层保存真相
- 同命令自动路由，减少模式命令膨胀
- 初始化集中配置，执行阶段尽量少打断
- 文档最小化，状态结构化，契约高价值化
- 默认自动推进，只有关键决策点暂停
- 中文目录与文档命名友好，内部协议保持稳定

## 5. 分层模型

### 5.1 插件层

插件仓库 `~/.claude/plugins/.../cx-workflow-marketplace/` 只负责通用能力:

- `skills/`
- `hooks/`
- `references/`
- `scripts/`
- `docs/`

插件层不保存任何具体项目的运行时状态，不再承担“当前 feature 做到哪”的职责。

### 5.2 项目层

项目级 `.claude/cx/` 是唯一运行时真相，保存:

- 项目配置
- 当前 feature 指针
- feature 级文档与状态
- fix 记录

### 5.3 规则层

项目中的 `AGENTS.md` / `CLAUDE.md` 只保留项目规范和最小约束，不再承载完整工作流实现细节。

## 6. 目录结构

未来项目级 `.claude/cx/` 建议收敛为:

```text
.claude/cx/
├── 配置.json
├── 状态.json
├── 功能/
│   └── 向量记忆/
│       ├── 需求.md
│       ├── 需求.json
│       ├── 设计.md
│       ├── 状态.json
│       ├── 任务/
│       │   ├── 任务-1.md
│       │   └── 任务-2.md
│       └── 总结.md
└── 修复/
    └── 六爻解卦500/
        └── 修复记录.md
```

约束:

- 根路径 `.claude/cx/` 保持不变
- 目录与文档命名允许中文
- 内部状态协议仍保留稳定英文字段
- feature 必须同时具备中文展示名和稳定 slug

示例:

- 目录名: `功能/向量记忆/`
- 内部 slug: `vector-memory`

## 7. 配置模型

### 7.1 配置真相

项目级 `配置.json` 是唯一公开配置源，负责:

- `developer_id`
- `github_sync`
- `current_feature`
- `agent_teams`
- `code_review`
- `auto_memory`
- `worktree_isolation`
- `auto_format`
- `hooks`

### 7.2 配置原则

- `config` 只描述项目偏好和运行分叉开关
- 不在 `config` 中保存任务级执行瞬时状态
- `current_feature` 仅作为恢复入口指针

### 7.3 公开配置瘦身

以下字段不再作为未来 `3.0` 的公开项目配置项:

- `batch_mode`
- `background_agents`
- `use_simplify`

这些属于实现策略，应由命令内部自动判断，而不是长期暴露给项目配置。

## 8. 状态模型

### 8.1 两层状态

- 根级 `状态.json`
  - 仅做项目摘要与索引
  - 记录初始化时间、最近活跃 feature、功能清单摘要

- feature 级 `状态.json`
  - 才是真正执行状态
  - 记录 `status`、`phases`、`tasks`、`depends_on`、`execution_order`

### 8.2 生命周期

每个 feature 统一走:

`drafting -> planned/executing -> blocked/completed -> summarized`

### 8.3 blocked 状态

`blocked` 是正式一等状态，不再只靠自然语言描述。

建议为阻塞原因保留结构化 `reason_type`，至少支持:

- `needs_decision`
- `missing_external_info`
- `dependency_unmet`
- `verification_failed`
- `environment_issue`
- `integration_conflict`

## 9. 命令面收敛

未来只保留以下命令:

- `/cx-init`
- `/cx-help`
- `/cx-status`
- `/cx-config`
- `/cx-prd`
- `/cx-plan`
- `/cx-exec`
- `/cx-fix`
- `/cx-summary`

不再发展 `tc` 平行命令面，也不新增 `quick/plan-lite/exec-lite` 作为一级入口。

## 10. 命令自动路由

### 10.1 cx-init

`cx-init` 是唯一一次重配置向导，负责:

- 初始化项目级 `.claude/cx`
- 单项目确认 `developer_id`
- 检查 GitHub remote
- 如无 remote，默认建议创建 GitHub 仓库并绑定
- 配置 `github_sync`
- 配置 `agent_teams`
- 配置 `code_review`
- 配置 `worktree_isolation`
- 配置 `auto_memory`
- 安装 hook 接线
- 补齐最小项目规则

初始化完成后，项目应直接可用。

### 10.2 cx-prd

职责:

- 收集并收敛需求
- 自动评估复杂度
- 自动判断是否需要 Design
- 通过勾选式问答让用户确认判断
- 写入 feature 级 `需求.md`

### 10.3 cx-plan

默认直接进入任务拆分。

仅当 `PRD` 明显引入新技术时，才触发:

- 技术识别
- skill 准备
- 外部文档查询

### 10.4 cx-exec

默认语义:

`/cx-exec = 自动推进当前 feature 的可执行任务，直到完成、阻塞或遇到关键决策点`

### 10.5 cx-fix

默认走快速修复路径，只有明显复杂时才升级为深度调试或多假设并行。

### 10.6 cx-summary

只负责闭环与同步，不承担执行中的状态推进职责。

## 11. 关键决策点

未来 `cx` 只应在以下四类场景暂停问询:

- 存在两条以上行为路径，且结果差异明显
- 会改动架构、API 契约、数据库结构、状态模型
- 涉及高风险或不可逆操作
- 需要用户提供外部信息

除此之外，默认自动推进，不应因普通实现细节频繁打断用户。

## 12. Agent Teams 模型

### 12.1 核心原则

Agent Teams 是内部编排机制，不是用户需要操心的操作界面。

### 12.2 默认模式

普通 `/cx-exec` 偏保守自治:

- 自动推进
- 可使用少量代理
- 不强制进入大规模团队编排

### 12.3 --all 语义

`/cx-exec --all` 的正式语义为:

`启用高自治、高并行的 agent teams 模式，按任务图自适应拉起 3+ 专业代理，尽可能推进当前 feature 的全部剩余任务。`

约束:

- 代理数量 `3+`，但由任务图自适应决定
- 仅当任务具备并行价值时才扩展团队
- 如果任务强依赖串行，不为“凑代理数”做假并行

### 12.4 角色策略

文档中只描述原则和示例，不写死固定编制。

常见角色示例:

- `backend-agent`
- `frontend-agent`
- `database-agent`
- `integration-agent`
- `review-agent`
- `investigator`

主代理职责收敛为:

- 读项目真相
- 切 wave 和分配任务
- 做契约校验与整合
- 在关键决策点暂停

## 13. Hook 策略

### 13.1 归属

hook 归插件层统一维护，运行时读取项目级配置与状态。

### 13.2 SessionStart

只做恢复摘要，不做长篇灌输。

### 13.3 PromptSubmit

从“固定频率提醒”收敛为“低噪声纠偏提醒”:

- 长会话漂移时提醒
- feature 中断恢复时提醒
- 无活跃 feature 时静默

### 13.4 PostEdit

只做轻量善后:

- 自动格式化
- 轻量状态刷新
- 可选变更摘要收集

### 13.5 PreCompact / Stop

只做保护，不做额外流程门槛。

### 13.6 通知

默认只在高价值节点通知:

- fix 完成
- feature 完成
- `--all` 大 wave 完成
- 关键决策点需要介入

## 14. 文档策略

未来 feature 级核心文档只保留:

- `需求.md`
- `设计.md`
- `任务/*.md`
- `总结.md`
- `修复记录.md`

原则:

- 状态信息优先结构化
- 文档优先服务执行
- `设计.md` 收敛为执行契约文
- `任务` 文件必须承载执行目标、依赖、改动范围、验收标准、契约片段
- `strategy.json` 不强制生成，仅在团队编排确有需要时生成

## 15. GitHub 集成

未来明确采用:

`项目内 .claude/cx 为主控真相`

`GitHub 为同步镜像与协作出口`

建议保留同步模式:

- `off`
- `local`
- `collab`
- `full`

但执行主线不再围绕 GitHub 状态机展开。

更适合承担 GitHub 交互的命令:

- `cx-init`
- `cx-plan`
- `cx-summary`

`cx-exec` 不应以 GitHub 作为主控中心。

## 16. 提交、验证与审查

### 16.1 提交粒度

默认每个 task 独立提交。

### 16.2 提交标记

commit message 统一要求在末尾带 feature 标记:

```text
type(scope): description [cx:<feature-slug>] [task:<n>]
```

fix 路径可使用:

```text
fix(scope): description [cx-fix:<fix-slug>]
```

### 16.3 验证策略

验证内建于 `cx-exec` / `cx-fix`:

- 优先跑当前改动最相关的验证
- 再视情况扩大范围
- 契约校验自动执行

### 16.4 审查策略

`code_review` 默认开启，但放在 feature 闭环后触发，不打断执行主线。

## 17. 失败与恢复

### 17.1 失败是正常路径

工作流默认承认以下场景会发生:

- 任务失败
- 测试失败
- 代理冲突
- 环境缺失
- 会话中断
- 外部服务不可用

### 17.2 自救优先

`cx-exec` / `cx-fix` 应先自救，再打断用户。

### 17.3 恢复顺序

恢复会话时，统一按以下顺序:

1. 读项目 `配置.json` 的 `current_feature`
2. 读对应 feature 的 `状态.json`
3. 定位 `blocked` / `in_progress`
4. 输出简短继续建议

恢复不应依赖模型上下文记忆。

## 18. 方案对比与结论

本次讨论中曾考虑三条路径:

1. 保守收敛，仅停用 `tc`
2. 纯 `cx 3.0` 内核化
3. 构建更大的兼容平台

最终选择:

`方案 2: 纯 cx 3.0 内核化`

理由:

- 最符合“以后只使用 cx”的目标
- 运行时真相清晰
- 可以承接 `tc` 的轻量体验而不保留双命令面
- 最有利于未来在所有项目中复用

## 19. 后续实施建议

建议按以下顺序落地:

1. 升级 `config-schema` 到 `3.0`
2. 定义新的项目级目录与状态协议
3. 改造 `cx-init`
4. 收敛 `cx-plan` 自动路由
5. 收敛 `cx-exec` 默认自治与 `--all` 语义
6. 收敛 `cx-fix` 模式升级逻辑
7. 调整 hook 为低噪声恢复模型
8. 更新 README / workflow-guide / skills 文档

## 20. 决策摘要

为方便后续实现，关键决策摘要如下:

- 只保留 `cx`，不再发展 `tc`
- 插件层放能力，项目层放真相
- feature 目录与文档允许中文命名
- 内部 JSON 字段与状态协议继续英文
- `current_feature` 指向稳定 slug，而非中文标题
- `cx-init` 一次性确认关键项目配置
- 默认自动推进，`--all` 进入高自治 agent teams 模式
- 只有四类关键决策点暂停问询
- GitHub 是同步镜像，不是主控真相
- 每个 task 独立 commit，末尾带 `cx` 标记
- `blocked` 状态与 `reason_type` 必须结构化
